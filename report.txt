Performance bottleneck report for optionlab/
Date: 2026-04-09

1. Brute-force implied volatility search on every option leg
Files:
- optionlab/engine.py:275
- optionlab/black_scholes.py:414-454
Why this is a bottleneck:
- `engine._run_option_calcs()` calls `get_implied_vol()` once per option leg.
- `get_implied_vol()` allocates a 1000-point volatility grid (`0.001 * arange(1, 1001)`), computes `d1`, `d2`, and a full Black-Scholes price vector for all 1000 candidates, then runs `argmin`.
- This is O(1000) Black-Scholes evaluations per leg even though only one scalar implied volatility is needed.
- For multi-leg strategies or repeated calls in screening/backtesting workloads, this dominates runtime and creates avoidable temporary arrays.
Justification:
- This is the clearest hotspot because it performs dense vector work for each leg regardless of how close the answer is. A root finder such as Brent/Newton with a bounded iteration count would reduce both CPU work and allocation pressure substantially.

2. Black-Scholes helper recomputes many expensive normal CDF terms per leg
Files:
- optionlab/engine.py:259
- optionlab/black_scholes.py:15-72
- optionlab/black_scholes.py:125-132
- optionlab/black_scholes.py:166-168
- optionlab/black_scholes.py:253-260
- optionlab/black_scholes.py:329-337
- optionlab/black_scholes.py:484-486
- optionlab/black_scholes.py:542-551
Why this is a bottleneck:
- `get_bs_info()` computes one option snapshot by calling many helpers in sequence.
- Those helpers repeatedly call `stats.norm.cdf(...)` on overlapping inputs (`d1`, `d2`, `-d1`, `-d2`, and related terms) instead of reusing already computed values.
- The function also recomputes shared transforms such as discount factors and exponentials in multiple helpers.
Justification:
- `stats.norm.cdf` is one of the more expensive scalar operations in this codebase. Repeating it several times per leg multiplies cost linearly with the number of strategy legs. Caching intermediate values inside `get_bs_info()` would reduce function-call overhead and repeated SciPy work.

3. Probability-of-profit calculations are repeated for the same strategy arrays
Files:
- optionlab/engine.py:197
- optionlab/engine.py:204-214
- optionlab/engine.py:216-224
- optionlab/support.py:201-247
- optionlab/support.py:509-610
- optionlab/support.py:428-459
Why this is a bottleneck:
- `engine._run()` always calls `get_pop()` once, and may call it up to two more times for `profit_target` and `loss_limit`.
- Each call recomputes profit/loss crossing detection, range construction, Black-Scholes probability integration, and expected-return aggregation over the same `stock_price_array` and `strategy_profit` arrays.
- On large price domains, these are full-array passes repeated two or three times.
Justification:
- This is avoidable repeated work. The stock-price grid and strategy P/L are already in memory, but range detection and probability integration are recomputed from scratch for each target. If users request both profit target and loss limit, runtime scales almost proportionally with the number of targets.

4. Non-business day counting performs a Python loop across every calendar day
Files:
- optionlab/utils.py:18-50
Why this is a bottleneck:
- `get_nonbusiness_days()` iterates day by day from `start_date` to `end_date`.
- For each date it builds `current_date`, calls `weekday()`, and formats the date with `strftime("%Y-%m-%d")` before checking holiday membership.
- This is pure-Python per-day work on a path used during input preparation for both target dates and option expirations.
Justification:
- The function is cached, which helps for repeated identical date ranges, but the first call for a long range is still O(number of days) with relatively expensive string formatting in the loop. Holiday membership should be checked with date objects directly, and weekend counting can be computed arithmetically rather than iterating each day.

5. Price grid generation can create very large cent-by-cent arrays
Files:
- optionlab/support.py:179-196
- optionlab/engine.py:48-51
- optionlab/engine.py:161-168
Why this is a bottleneck:
- `create_price_seq()` builds the stock-price domain at $0.01 resolution for the entire `[min_stock, max_stock]` range.
- The array length is `(max_price - min_price) * 100 + 1`, so broad price intervals produce very large arrays.
- Downstream functions allocate profit matrices sized by this grid, so one oversized domain expands memory use and compute cost everywhere.
Justification:
- This is a multiplicative bottleneck: the dense grid affects every profile, PoP pass, and plotting operation. A user-supplied step size, adaptive resolution, or a cap on grid size would prevent large accidental slowdowns and memory spikes.

6. Engine preallocates full profit matrices for every leg across the full stock grid
Files:
- optionlab/engine.py:161-168
- optionlab/engine.py:171-182
Why this is a bottleneck:
- `_run()` allocates `data.profit` with shape `(number_of_legs, number_of_stock_prices)` and then also accumulates `strategy_profit`.
- In array mode it allocates a second full matrix `profit_mc`.
- The algorithm then processes one leg at a time, but still keeps the entire per-leg matrix in memory.
Justification:
- This increases memory bandwidth and allocation cost, especially because the stock grid can already be large. If callers do not need all per-leg profit arrays simultaneously, parts of this could be streamed or stored more selectively.

Priority order:
1. Brute-force implied volatility search
2. Repeated PoP computations for multiple targets
3. Repeated Black-Scholes normal CDF work
4. Dense cent-by-cent price grid expansion
5. Full-matrix profit allocation
6. Day-by-day non-business date counting

Notes:
- The report focuses on performance bottlenecks only, not correctness issues.
- The line references above are the places where the expensive work is initiated or implemented.
